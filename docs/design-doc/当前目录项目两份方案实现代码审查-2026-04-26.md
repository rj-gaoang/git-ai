# 当前目录项目两份方案实现代码审查报告

> 审查日期：2026-04-26
> 审查范围：`git-ai`、`spec-kit`、`ai-cr-manage-service`
> 对照方案：
> 1. 《格式化或纯空白改动不触发归因切换的方案》
> 2. 《git-ai看板方案》

## 结论

两份方案的主链路大部分已经落到了代码上，但并没有形成完全闭环。

- `git-ai` 的“格式化/纯空白改动不触发归因切换”主逻辑已经实现，包含 feature flag、段对识别、superblock 预扫描以及集成测试文件。
- `git-ai` 的 commit 后自动上传、`spec-kit` 的 `post-init` 自动安装、PowerShell 上传脚本、Code Review 模板扩展也已经落地。
- 真正的缺口集中在 `ai-cr-manage-service` 与客户端协议的对齐层，以及 `spec-kit --script sh` 的跨平台补齐层：目前这些问题会导致看板筛选失真、提交级工具统计丢失、批量上传结果被误报成功，以及 bash 项目无法按设计执行上传流程。

## 主要发现

### P1：服务端把 `reviewDocumentId` 当成 `hasAuthorshipNote` 使用，导致“有/无 note”筛选与详情展示失真

**结论**：该功能没有按设计实现完成，当前看板里的 `has-note` / `no-note` 语义是不可靠的。

**证据链**：

- 客户端已经明确上传 `hasAuthorshipNote`
  - `spec-kit/scripts/powershell/upload-ai-stats.ps1:1332`
  - `git-ai/src/integration/upload_stats.rs:174`
- 服务端接收 DTO `GitAiStatsAddCommitDTO` 中没有 `hasAuthorshipNote` 字段
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/domain/dto/GitAiStatsAddCommitDTO.java:20-56`
- 服务端落库实体 `GitAiCommitStats` 也没有对应字段
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/domain/po/GitAiCommitStats.java:24-92`
- 看板筛选与详情页改用 `review_document_id` 充当“是否有 note”的判断依据
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/service/impl/GitAiStatsServiceImpl.java:580-583`
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/mapper/GitAiCommitStatsMapper.java:117-132`
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/mapper/GitAiSummaryStatsMapper.java:32-54`

**实际影响**：

- 手动上传场景里，即使 commit 有真实 authorship note，只要 `reviewDocumentId` 为空，就会被错误归到 `no-note`。
- Code Review 场景里，只要顶层携带了 `reviewDocumentId`，即使某些 commit 实际没有 note，也会被错误归到 `has-note`。
- 这会直接破坏第二份方案里看板与审查报告对“note 有无”的判断。

**建议**：

- 在 commit 维度新增并落库 `has_authorship_note` 字段。
- 服务端筛选、详情页、聚合页统一改为基于 commit 级 `hasAuthorshipNote` 判断，而不是复用 `reviewDocumentId`。

### P1：提交级 `toolModelBreakdown` 与服务端 DTO 层级不一致，提交级工具统计会被静默丢失

**结论**：该功能只实现了“文件级工具统计”，没有完整实现“提交级工具统计”。

**证据链**：

- `spec-kit` 上传脚本把工具拆解放在 `stats.toolModelBreakdown` 下，而不是 commit 顶层
  - `spec-kit/scripts/powershell/upload-ai-stats.ps1:1327-1333`
  - `spec-kit/scripts/powershell/upload-ai-stats.ps1:1258-1272`
- `git-ai` 原生自动上传同样把 `toolModelBreakdown` 放在 `stats` 内部
  - `git-ai/src/integration/upload_stats.rs:174-183`
  - `git-ai/src/integration/upload_stats.rs:186-218`
- 服务端却从 commit 顶层读取 `commitDTO.getToolModelBreakdown()`
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/service/impl/GitAiStatsServiceImpl.java:124-127`
- DTO 也把 `toolModelBreakdown` 定义在 commit 顶层，而不是 `stats` 内
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/gitai/domain/dto/GitAiStatsAddCommitDTO.java:51-56`

**实际影响**：

- 服务端当前不会拿到提交级工具/模型统计。
- 看板里按 commit 维度展示“主要工具/模型”时，结果会缺失或回退成文件级、副作用字段或空值。
- 第二份方案里强调的 `toolModelBreakdown[]` 兼容只做到了文件级，没有打通 commit 级。

**建议**：

- 二选一统一协议层级：
  - 要么客户端把 `toolModelBreakdown` 提升到 commit 顶层；
  - 要么服务端改为从 `commitDTO.getStats().getToolModelBreakdown()` 读取，并同步调整 DTO。

### P1：批量上传接口没有返回 `results[]`，客户端会把“无逐条结果”的响应默认解释为全部成功

**结论**：第二份方案里的“批量逐条回执”没有完整实现，当前只能得到“整批成功/失败”的弱语义。

**证据链**：

- `spec-kit` 上传脚本在服务端未返回 `results/commits/items` 时，直接把整批所有 commit 标成 `uploaded`
  - `spec-kit/scripts/powershell/upload-ai-stats.ps1:1358-1368`
- 服务端公共上传接口返回的是 `R<Void>`，并非方案设计里的 `results[]`
  - `ai-cr-manage-service/ruijie-codereview-manage/src/main/java/com/ruoyi/codereview/document/controller/CodereviewApiPublicDataController.java:143-145`
  - `ai-cr-manage-service/ruoyi-common/src/main/java/com/ruoyi/common/core/domain/R.java:14-83`

**实际影响**：

- 如果一批 commit 中存在部分失败、部分重复、部分落库异常，客户端仍会把所有 commit 显示为成功上传。
- Code Review 自动补传场景无法精确提示哪一条失败。
- 这与第二份方案设计的 `results[]` 逐条状态完全不一致。

**建议**：

- 服务端补齐批量返回结构，例如 `total/succeeded/failed/results[]`。
- 客户端只有在明确收到某个 commit 的成功状态时，才把该 commit 计为成功上传。

### P2：`--script sh` 路径没有上传脚本，Code Review 模板却硬编码调用 PowerShell 脚本

**结论**：第二份方案在跨平台实现上仍然是部分完成状态。

**证据链**：

- 仓库存在 `spec-kit/scripts/bash/post-init.sh`，说明 `sh` 路径是受支持的
  - `spec-kit/scripts/bash/post-init.sh`
- 但仓库中不存在 `spec-kit/scripts/bash/upload-ai-stats.sh`
  - 文件检索结果：未找到 `spec-kit/scripts/bash/upload-ai-stats.sh`
- Code Review 模板固定要求执行 `.specify/scripts/powershell/upload-ai-stats.ps1`
  - `spec-kit/templates/commands/code-review.md:673`

**实际影响**：

- 使用 `specify init --script sh` 初始化的项目，没有与设计匹配的上传脚本。
- Code Review 流程在 bash 项目中会指向一个不存在的文件路径。

**建议**：

- 补齐 `scripts/bash/upload-ai-stats.sh`；或
- 明确限制上传能力仅支持 `ps`，并同步修改模板与文档，不要在 `sh` 路径中继续暴露这条能力。

### P3：第一份方案中的 metrics 可观测性未完整落地，目前只看到了 debug 输出

**结论**：格式化透传的功能主逻辑已实现，但方案里承诺的 metrics 计数器没有看到对应实现。

**证据链**：

- 已实现内容：
  - feature flag：`git-ai/src/feature_flags.rs:61-62`
  - 段对与 superblock 逻辑：`git-ai/src/authorship/attribution_tracker.rs:900-1159`
  - 核心判断与超块扫描：`git-ai/src/authorship/attribution_tracker.rs:1931-2027`
  - 集成测试文件：`git-ai/tests/integration/format_only_attribution.rs`
- 已看到 debug 输出：
  - `git-ai/src/authorship/attribution_tracker.rs:1156-1159`
- 未检索到方案中提到的 `attribution.format_only.segments` / `attribution.format_only.lines_preserved` 计数器实现。

**实际影响**：

- 功能本身可用，但上线后如果出现误判或回归，缺少方案里承诺的可观测指标，排查成本会偏高。

**建议**：

- 如果该方案准备进入稳定使用阶段，补齐指标埋点；否则至少把“metrics 暂未实现”写进方案验收状态，避免误判为已完成。

## 建议的修复优先级

1. 先修服务端 `hasAuthorshipNote` 落库与查询逻辑，这是看板语义错误，影响最直接。
2. 统一 commit 级 `toolModelBreakdown` 协议层级，否则提交维度工具统计一直不可信。
3. 补齐服务端批量 `results[]` 返回，避免客户端把“无逐条回执”误判为全量成功。
4. 明确处理 `--script sh` 路径：要么补 bash 上传脚本，要么从模板与文档中移除该能力承诺。
5. 视上线需求决定是否补齐格式化透传的 metrics 埋点。
