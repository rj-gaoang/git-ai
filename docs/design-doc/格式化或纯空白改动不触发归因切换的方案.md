> 日期：2026-04-24
> 作者：方案规划
> 关联模块：`git-ai/src/authorship/attribution_tracker.rs`

---

## 1. 背景与问题

### 1.1 当前归因模型回顾

git-ai 的字符级归因核心在 [git-ai/src/authorship/attribution_tracker.rs](git-ai/src/authorship/attribution_tracker.rs)，核心规则可以概括为：

1. 用 `imara_diff`（Myers 算法）按字节比对 `before` 与 `after` 内容。
2. 对每个 diff 操作（Equal / Delete / Insert）做归因转换：
   - **Equal**：原归因按位置平移到新内容。
   - **Delete**：人类 checkpoint 下，纯空白删除不会留下"作者标记"；AI checkpoint 下任何删除都会留标记。
   - **Insert**：人类 checkpoint 下若是"前一段是纯空白删除 + 当前是纯空白插入"（`is_formatting_pair`），会**继承上一条归因**而不是改判为人类；其它非空白插入都按 `current_author + 当前 ts` 标记。
3. 字符级归因再聚合成行级归因（`find_dominant_author_for_line_candidates`），规则是：
   - 在同一行有重叠的归因里，**时间戳最新者获胜**；
   - AI 与 Human 同时存在且 Human 较新，则 `overrode = ai_author_id`，本行算"人类覆盖了 AI"。

### 1.2 痛点：仅格式化也会改变归因

虽然源码里已经针对"成对的空白删除 + 空白插入"做了 `is_formatting_pair` 处理，但**实际生产中仍存在多种格式化场景没有被覆盖**：

1. **纯缩进调整（如 Tab → 空格、2 空格 → 4 空格）**
   单行格式化只改空白时，部分边缘情况会被识别为非配对的"重写"——一行内既有空白又有非空白被替换，diff 库可能把它输出为多个交错的 Insert/Delete，破坏 `is_formatting_pair` 的成对前提。

2. **跨多行的格式化器（Prettier / rustfmt / Black / clang-format）**
   格式化工具会一次性重写大段缩进、换行、尾随空白。`imara_diff` 的字节级输出常常是"删除一大块 + 插入一大块"，删除段和插入段都包含**少量非空白字符**（被移动过的标识符、括号），这就让 `data_is_whitespace()` 判 false，归因被改写到当前作者。

3. **行尾换行符 / EOL 转换（CRLF ↔ LF）**
   只改换行符的提交，会被当成全文重写，整文件归因可能被吞掉。

4. **末尾空白裁剪（trim trailing whitespace）**
   保存时编辑器自动裁剪行尾空格，归因被人类 checkpoint 覆盖。

5. **连续空行合并 / 增删空行**
   空行被视为"插入了换行符"（`contains_newline = true`），按现有规则会强制使用 `current_author`，从而把整段空行归到操作者。

### 1.3 用户预期

> "代码块是 AI 写的，我只是按了一下『格式化文件』，归因就变了。"

期望行为：**当一个变更在『语义上等价』（即去掉所有空白后内容完全相同）时，相关行/字符不应改变归因。**

## 2. 设计目标

| 目标 | 描述 |
|------|------|
| G1 | 纯空白/纯换行的变更不改变字符级归因。 |
| G2 | 格式化器导致的"内容相同但分布不同"的多行重写，归因继承自原作者。 |
| G3 | 不影响真正的语义编辑：只要有非空白字符增删或修改，仍按现有规则归因。 |
| G4 | 行级归因保持单调一致：不会因为格式化导致一行从 AI 变 human、再变回 AI 抖动。 |
| G5 | 实现成本可控，不引入语言相关解析（不要 AST，纯字符级）。 |
| G6 | 可观测、可关闭（feature flag / config）。 |

非目标：

- 不打算理解语法层面的"等价"（比如重命名变量）。
- 不打算做语言感知的 token diff。

## 3. 当前实现的不足分析

集中在 `update_attributions_for_checkpoint()` 函数（~880-1100 行）：

### 3.1 `is_formatting_pair` 的触发条件过窄

```rust
let is_formatting_pair = prev_whitespace_delete && is_whitespace_only;
```

只有满足"上一段是纯空白删除 + 当前段是纯空白插入"才生效。现实中的格式化往往会输出：

```
Delete  "    fn foo() {\n        bar();\n    }"   ← 含非空白
Insert  "  fn foo() {\n    bar();\n  }"           ← 含非空白
```

整段 Delete/Insert 都不是"纯空白"，于是 `data_is_whitespace()` 返回 false，归因被覆盖。

### 3.2 没有"段对齐"机制

diff 算法把变更切分为多个交错的 Equal / Delete / Insert 段。我们目前**逐段独立判断**，没有在一个更大的上下文里识别"这一组 Delete + Insert 的去空白结果是相等的"。

### 3.3 行级聚合按时间戳硬胜

`find_dominant_author_for_line_candidates()` 完全按 `ts` 选获胜者，没有"如果非空白内容没变则保留旧作者"的兜底。

## 4. 设计方案

整体分成三层防御，越靠下越精确，但都使用相同的判断函数 `is_format_only_change(old, new) -> bool`，以保证一致性。

### 4.1 核心判断函数：`is_format_only_change`

```rust
/// 判断两段字节序列是否仅在空白上有差异（包括空格、Tab、CR、LF、其他 Unicode 空白）。
/// 返回 true 表示"格式化等价"，即去掉所有 ASCII/Unicode 空白后内容完全相同。
fn is_format_only_change(old: &[u8], new: &[u8]) -> bool {
    iter_non_whitespace(old).eq(iter_non_whitespace(new))
}
```

要点：

- 用迭代器对比，避免分配新 String。
- 针对 UTF-8 安全：以 `char` 为单位比较，跳过 `char::is_whitespace()` 为 true 的字符。
- 复杂度 O(n)。

### 4.2 第一层：扩大 `is_formatting_pair` 的判定范围（段对）

对每对相邻的 `Delete + Insert` 操作，**额外计算** `is_format_only_change(delete_data, insert_data)`：

```
旧逻辑：
  is_formatting_pair = prev_whitespace_delete && is_whitespace_only

新逻辑：
  is_formatting_pair =
       (prev_whitespace_delete && is_whitespace_only)            // 旧分支保留
    || is_format_only_change(prev_delete_data, insert_data)      // 新分支
```

实现细节：

1. 在循环顶部新增 `let mut prev_delete_data: Option<&[u8]> = None;`
2. 进入 Delete 分支时，缓存 `prev_delete_data = Some(diff.data());`
3. 进入 Insert 分支时，调用 `is_format_only_change`。
4. 进入 Equal 分支时，清空 `prev_delete_data`（断开成对关系）。

这样可以吃掉绝大多数"逐行格式化"和"段格式化"的情况。

### 4.3 第二层：跨段聚合检测（多段 Delete/Insert 交错）

格式化器有时会输出形如：

```
Delete A1
Insert B1
Equal  E1
Delete A2
Insert B2
```

而真正的判断应该是 `concat(A1, A2) ≡ concat(B1, B2)`（去空白等价）。

提案：

1. 在做归因前先扫描一遍 diff 列表，把**连续不被 Equal 分隔的 Delete/Insert**收集到一个"超块（superblock）"。
2. 对每个超块整体调用 `is_format_only_change`。
3. 整个超块判定为"format only"时，标记其内每个 Delete/Insert 段都走"继承归因"路径（与 `is_formatting_pair` 相同）。

实现位置：在主循环之前增加一次 O(n) 预扫描，把超块结果存入 `HashMap<deletion_idx | insertion_idx, bool>`。主循环中检索此表，优先于现有 `is_formatting_pair`。

### 4.4 第三层：行级聚合的"语义不变"兜底

一句话解释：**前两层（4.2 / 4.3）是在字符级 diff 阶段尽量保住原归因；4.4 是最后一道保险，专门防止“字符级已经大体保住了，但到了按行汇总时，仍因为最新时间戳胜出而把整行错误改判”。**

也就是说，4.4 不是新的主流程，而是一个可选兜底：

- 如果 4.2 / 4.3 已经足够解决格式化误归因，就不需要做 4.4。
- 只有当我们上线 4.2 / 4.3 后，仍发现某些“非空白内容没变、但整行作者被改掉”的边缘案例，才需要加这一层。

一个更直观的例子：

- 原来这一行是 AI 写的：`const x = foo(a, b);`
- 你只做了格式化，变成：`const x = foo(a,b);` 或因为换行策略变化被重新折行
- 理论上字符级阶段已经尽量把旧归因保住
- 但如果行级汇总时仍因为“这次 human checkpoint 的时间戳更新了某些空白片段”而让整行改判为 human
- 那 4.4 就会说："这行去掉空白后和上一版本其实一样，那整行仍沿用上一版主导作者，不要改判"

在 `find_dominant_author_for_line_candidates()` 里增加一条规则：

```
如果调用方传入 prev_line_content 与本次 line_content 满足
  is_format_only_change(prev_line_content, line_content) == true
那么：
  优先返回 prev_line_author（如果存在）
  并设置 overrode = None
```

这要求把"上一版本同一逻辑行"的内容传进来。可以通过：

- 上层 `attributions_to_line_attributions_for_checkpoint` 接受额外参数 `prev_line_index_map: Option<&LineIndex>`；
- 在 Equal 段里基于行号偏移构造该映射；
- 没有映射的行（真正新增的）走原逻辑。

> 注：第三层是**保险**层，第一/二层覆盖到位时多半不会触发，但它能兜住极端情况。
> 如果实现复杂度太高，可分阶段落地：先做 4.2 / 4.3，第三层延后。
>
> **本方案的默认建议**：4.4 不纳入第一版实现，不需要当前拍板；先按 4.2 / 4.3 落地，再根据测试和真实样本决定是否补上这一层。

### 4.5 行内继承的归因 ts 处理

继承归因时**不要更新 ts**：

```rust
// 之前
(attr.author_id.clone(), ts)            // ❌ 用了当前时间戳

// 之后
(attr.author_id.clone(), attr.ts)       // ✅ 保留原时间戳
```

这能保证后续的"AI 后于人类 / 人类后于 AI"判定不被格式化操作扰动。
现有 `is_formatting_pair` 已经是这样写的，新增分支也必须严格遵守。

### 4.6 配置开关

新增一个 feature flag（参见 `src/feature_flags.rs` 的 `define_feature_flags!` 宏）：

```rust
format_only_attribution_passthrough: bool = true (release) | true (debug);
```

外部覆盖：环境变量 `GIT_AI_FORMAT_PASSTHROUGH=0/1`，便于排查回归时快速回退。

### 4.7 可观测性

- 在 `update_attributions_for_checkpoint` 末尾，如果至少有一段被判定为"format only inherit"，通过 `debug_log!` 输出：

  ```
  [git-ai] format-only passthrough: file=src/foo.rs segments=3 inherited_authors=[copilot/abc, human]
  ```

- 在 metrics 里新增计数器：
  - `attribution.format_only.segments`（被判格式化的 Delete/Insert 段数）
  - `attribution.format_only.lines_preserved`（行级兜底命中数）

## 5. 边界与陷阱

| 场景 | 处理 |
|------|------|
| 字符串字面量内的真正空格变化（如 `"a b"` → `"a  b"`） | 仍判定为 format-only，**保留原作者**——这是有意识的取舍。语义层判定超出本方案目标。需要在文档里写明。 |
| 注释内容修改但只动了空白 | 同上，按格式化处理。 |
| Markdown / YAML / Python 等空白敏感语言 | 修改缩进会真的改变语义，但归因层不应擅自承担"代码语义有效性"的责任；保留原作者更接近用户直觉（"我没改逻辑，只调缩进"）。 |
| 行尾空白裁剪导致全文 diff | 整体被识别为超块，判格式化，全部继承原作者。✅ 期望。 |
| 删除整段空行 / 增加整段空行 | 当前 `contains_newline → current_author` 的硬规则要修改为：先看是否处于"format-only 超块"，是则继承。 |
| AI checkpoint 触发的格式化 | 当前 AI 路径有 `if is_ai_checkpoint { current_author }` 的硬路径，**保持不变**——AI 主动重写仍归 AI，符合现有产品决策（参见 src 注释 line 591）。新机制只对 Human / KnownHuman checkpoint 启用。 |
| Move detection 已识别为移动 | 优先级最高，本方案不动 move 路径。 |

### 6.2 主循环的伪代码草图

```rust
fn update_attributions_for_checkpoint(...) -> Vec<Attribution> {
    // ... 现有 move detection、deletion_to_move 构造 ...

    // ★ 新增：预扫描格式化超块
    let format_only_segments = if feature::format_only_passthrough() {
        scan_format_only_superblocks(diffs)
    } else {
        Default::default()
    };

    let mut prev_delete_data: Option<&[u8]> = None;

    for diff in diffs {
        match diff.op() {
            ByteDiffOp::Equal => {
                // ... 现有逻辑 ...
                prev_delete_data = None;
            }
            ByteDiffOp::Delete => {
                if format_only_segments.contains(&deletion_idx) {
                    // 跳过零长度 marker 注入
                } else {
                    // 现有 is_ai_checkpoint || !data_is_whitespace 分支
                }
                prev_delete_data = Some(diff.data());
                // ...
            }
            ByteDiffOp::Insert => {
                let is_format_pair_legacy = prev_whitespace_delete && is_whitespace_only;
                let is_format_pair_new = prev_delete_data
                    .map(|d| is_format_only_change(d, diff.data()))
                    .unwrap_or(false);
                let is_format_superblock = format_only_segments.contains(&insertion_idx);

                let inherit = is_format_pair_legacy
                    || is_format_pair_new
                    || is_format_superblock;

                if inherit {
                    // 与现有 is_formatting_pair 分支同——继承上一条 attribution
                    // 注意：保留原 ts
                } else {
                    // 现有 current_author + ts 分支
                }
                prev_delete_data = None;
            }
        }
    }
}
```

## 8. 风险与回滚

| 风险 | 缓解 |
|------|------|
| 误判：把真实的字符串字面量改动当成格式化 | 文档里明确该取舍；提供 feature flag 一键关闭 |
| 性能：超块扫描多一次 O(n) 遍历 | 单文件 diff 通常很小；feature flag 关闭 = 零开销 |
| 与 Move detection 冲突 | Move 优先；超块扫描跳过已经在 `deletion_to_move` 里的 deletion_idx |
| 与 AI checkpoint 的"重写归 AI"行为冲突 | 新机制仅对 Human / KnownHuman 启用 |
| 行级兜底（4.4）实现复杂 | 分阶段：先上 4.2 + 4.3，行级兜底独立 PR |

回滚路径：

- 第一阶段：`GIT_AI_FORMAT_PASSTHROUGH=0` 即可临时禁用。
- 第二阶段：去除 feature flag，把行为做成默认。

    - 决策结果：第一版先不做

   - 当前结论：否（现有产品决策"AI 重写归 AI"应保留）。如有用户反馈再讨论。
